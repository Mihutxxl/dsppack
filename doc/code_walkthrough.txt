================================================================================
        DSPPACK - FILTER DESIGN AND ANALYSIS TOOL FOR GNU OCTAVE
                    LINE-BY-LINE CODE EXPLANATION
================================================================================

Project: dsppack (GNU Octave package)
Purpose: A graphical tool for designing, analysing, and comparing digital
         filters (IIR and FIR) with interactive pole-zero editing, audio
         demonstration, code export, and report generation.

The inst/ directory contains all runnable Octave source files.
Each file is explained below with a line-by-line commentary.

================================================================================
FILE 2: dsp_compute_filter.m     (Filter Coefficient Computation)
================================================================================

This function takes the numeric parameters from the GUI and returns the
numerator (b) and denominator (a) transfer function coefficients.

Lines 1-10:  Texinfo help block documenting all parameters:
             Fs=sampling freq, Fc/Fc2=cutoffs, N=order, type_idx selects
             lowpass/highpass/bandpass/bandstop, topo_idx selects the design
             method, arch_idx selects IIR or FIR, Rp/Rs are ripple/attenuation.

Line 12:     Function signature: [b, a] = dsp_compute_filter(...).

Lines 13-15: Default argument handling.
             If arch_idx not given, default to 1 (IIR).
             If Rp not given or empty, default to 3 dB.
             If Rs not given or empty, default to 40 dB.

Lines 17-18: type_strs maps the numeric type index to a string:
             1="low", 2="high", 3="bandpass", 4="stop".  These strings
             are required by Octave's filter design functions.

Lines 20-24: Compute the normalised cutoff frequency Wn.
             For low/high pass (type_idx <= 2): Wn = Fc / (Fs/2).
             For bandpass/bandstop: Wn = [Fc, Fc2] / (Fs/2).
             Division by Fs/2 normalises to the Nyquist frequency.

Lines 26-29: Validate Wn.  All values must be strictly between 0 and 1
             (exclusive).  If not, an error is raised.

Lines 31-42: IIR filter design (arch_idx == 1).
             Uses a switch statement on topo_idx:
               case 1: butter(N, Wn, type_str) -- Butterworth design.
               case 2: cheby1(N, Rp, Wn, type_str) -- Chebyshev Type I.
               case 3: cheby2(N, Rs, Wn, type_str) -- Chebyshev Type II.
               case 4: ellip(N, Rp, Rs, Wn, type_str) -- Elliptic (Cauer).
             Each returns [b, a] polynomial coefficient vectors.

Lines 43-69: FIR filter design (arch_idx == 2).
             Line 45: a = 1 because FIR filters have no feedback (denominator
                      is always 1).
             Lines 47-60: Window-based FIR design (topo_idx 1-4).
               Lines 49-53: Generate the window function of length N+1:
                            hamming, hanning, blackman, or kaiser(N+1, 5).
               Lines 54-60: Call fir1(N, Wn, ..., win) with the appropriate
                            type specification.
             Lines 61-68: Optimal FIR design (topo_idx 5-6).
               Line 62: make_pm_bands builds frequency/amplitude band vectors.
               Line 64: firpm(N, fb, ab) -- Parks-McClellan algorithm.
               Line 66: firls(N, fb, ab) -- Least-squares method.

Lines 72-100: make_pm_bands helper function.
              Converts cutoff frequency Wn and type index into frequency (f)
              and amplitude (amp) vectors needed by firpm/firls.
              Line 73: gap defines a transition band width (at least 0.05).
              Lines 74-91: Switch on type_idx to build the band specification:
                case 1 (LP): passband [0..Wn-gap], stopband [Wn+gap..1].
                case 2 (HP): stopband [0..Wn-gap], passband [Wn+gap..1].
                case 3 (BP): stop-pass-stop three-band structure.
                case 4 (BS): pass-stop-pass three-band structure.
              Lines 95-99: Post-processing loop ensures the frequency vector
                           is strictly increasing (required by firpm/firls).
                           If any value is not greater than its predecessor,
                           it is nudged up by 0.001.

================================================================================
FILE 5: dsp_check_stability.m     (BIBO Stability Analysis)
================================================================================

Analyses whether a discrete-time filter is BIBO (Bounded-Input Bounded-Output)
stable by examining whether all poles lie inside the unit circle.

Lines 1-23:  Texinfo documentation describing all returned struct fields.

Line 25:     Function declaration: s = dsp_check_stability(b, a).

Line 26:     tol = 1e-10 defines a numerical tolerance to handle floating-point
             imprecision near the unit circle boundary.

Line 28:     roots(a) computes the poles of the filter (roots of the
             denominator polynomial).

Line 29:     abs(s.poles) gives the magnitude (radius) of each pole in the
             z-plane.

Lines 31-37: Handle the FIR case.  If there are no poles (a = [1]), radii
             is empty.  The filter is trivially stable with maximum radius 0,
             margin 1, and status "STABLE (FIR)".

Line 39:     max_radius = max(s.radii) finds the pole closest to (or outside)
             the unit circle.

Line 40:     margin = 1 - max_radius measures how far inside the unit circle
             the worst-case pole sits.  Positive = stable, negative = unstable.

Lines 42-51: Classify stability:
             - max_radius < 1 - tol: all poles strictly inside -> STABLE.
             - max_radius <= 1 + tol: on or very near the boundary ->
               MARGINALLY STABLE (could oscillate without growing/decaying).
             - max_radius > 1 + tol: at least one pole outside -> UNSTABLE.

================================================================================
FILE 7: dsp_compute_f3db.m     (-3 dB Frequency Computation)
================================================================================

Finds where the magnitude response drops 3 dB below its peak.  This is the
standard "half-power" bandwidth measure.

Lines 1-7:   Texinfo documentation.

Line 8:      Function declaration: f3db_str = dsp_compute_f3db(H, f).

Line 9:      Convert complex response H to decibels.

Line 10:     Find the peak magnitude in dB.

Line 11:     Find sign-change crossings of the curve (mag_db - (max_db - 3)).
             diff(sign(...)) is non-zero where the curve crosses the -3 dB
             threshold.

Lines 12-18: Format the result.
             If crossings exist, report the first crossing frequency.
             If there are multiple crossings (e.g., bandpass has two), also
             report the last crossing.
             If no crossings found, return "N/A".

    |     +-- dsp_check_stability.m
    |     +-- dsp_compute_f3db.m     -3 dB frequency finder
    |
    +-- dsp_compare_window.m     Side-by-side filter comparison
    |     +-- dsp_check_stability.m
    |     +-- dsp_compute_f3db.m
    |
    +-- dsp_export_callback.m    Exports coefficients to workspace
    +-- dsp_code_export.m        Exports as Octave/C/Python code
    +-- dsp_save_callback.m      Saves design to .mat file
    +-- dsp_load_callback.m      Loads design from .mat file
    +-- dsp_generate_report.m    Generates HTML report with embedded plots
    +-- dsp_audio_demo.m         Audio loading, filtering, playback, spectrograms
